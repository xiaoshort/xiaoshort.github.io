什么是MySQL？

​ MySQL是一种关系型数据库管理系统，所谓关系型指的是建立在关系模型基础上，采用集合代数等数学概念和方法集中处理数据库中数据的模式。

MySQL有哪些数据库类型？

分为数值型、字符串型、日期时间型

  • 数值型

    ​ 整数型：tinyint（1bt）、smallint（2bt）、medimint（3bt）、int（4bt）、bigint（8bt）

    ​ 小数型：float、double、decimal

  • 字符串型

    ​ char、varchar、text、blob

  • 日期时间型

    ​ datetime、date、timestamp

varchar和char有什么区别，如何进行选择？

区别：

  • 存储长度上：char是定长的，范围在0255之间，varchar是变长的，范围在064k之间
  • 存储效率上：由于char是定长的，存储效率高于varchar
  • 存储方式上：char对英文字符占1字节，中文2字节，varchar均为2字节

优缺点：

​ char：存储空间固定，存储读取速度快，但容易造成空间冗余

​ varchar：变长的，节约空间，但读取和存储时候，需要读取信息计算下标，才能获取完整内容

选择：

  • 对于固定长度或者字段长度较小的属性，用char
  • 对于字段较长，长度不固定的属性，用varchar
  • 在第二点的前提下，如果属性经常需要变更，为了防止使用varchar的额外计算，拖慢效率，用char

什么是数据库设计的范式和反范式？

  • 范式：在关系数据库中构造数据库必须遵循一定的规则，这种规则就是范式，范式可以减少数据冗余，节约存储空间，更新操作快，但查询效率低，更难进行索引优化
  • 反范式：通过数据冗余减少数据表之间的关联性，提高查询性能和索引支持化，但浪费存储空间，数据不易维护

​

什么是索引，为什么要使用索引，常见的索引类型有哪些？

  • 索引是帮助MySQL高校获取数据的数据结构，就好像书本的目录一样

  • 在数据量比较大的情况下，只通过遍历的查询效率低，需要引入索引，提高数据检索效率，降低数据库io成本和cpu消耗（但会占用磁盘空间，降低更新效率）

  • 索引按应用层面划分，常见的有：

    ​ 主键索引（唯一不为空）

    ​ 唯一索引（唯一可为空）

    ​ 普通索引（不唯一可为空）

    ​ 全文索引（仅限字符串型，针对字段较长的属性）

怎样科学合理地设计数据库索引？

  • 索引需要占据磁盘空间，且自身存在性能消耗，因此不建议小表使用索引，建议数据量大且经常需要查询的表使用索引
  • 建立索引时，选择经常作为查询条件、排序分组联合依据的字段建立索引
  • 选择唯一性索引保证数据的唯一性，并限制索引数量

索引的数据类型是什么(InnoDB)，为什么采用它？

B+树

  • 与Hash索引相比
    • hash索引建立后无法与原顺序保持一致，因此无法进行范围查询、模糊查询和排序操作，且当数据量大时可能产生hash碰撞，从而降低查询效率
    • B+索引支持范围查询和排序，查询效率稳定（根据树高决定）
  • 与普通二叉树相比
    • 二叉树会随着数据量的增加而急剧增加树高，导致查询效率变低
    • 而B+树可以显著减少树高，从而提高查询效率
  • 与B树相比
    • B树的叶子节点和非叶子节点都存储数据，磁盘io读取的次数更多，效率较低
    • B+树数据只存储在叶子节点，磁盘io读取的次数更少，效率更高，且B+树结合链表有序连接，对于范围查询，B+树的链表遍历比B树的中序遍历效率更高

什么是回表查询，什么情况下不会进行回表查询操作？

​ 在InnoDB引擎中，非主键索引查找数据时，需要先根据非主键索引查找主键，再根据主键查找行数据，这种再查找的行为称为回表查询

​ 当存在索引覆盖时，不会进行回表查询，即，将select查询语句的字段全部添加到联合索引中，不会走回表查询（开发中很少遇到）

简述InnoDB引擎与MyISAM引擎的区别？

  • 从功能方面

    ​ InnoDB支持事务，外键，行锁和MVCC(多版本并发控制)模式的读写

    ​ MyISAM不支持事务，但支持全文索引

  • 从效率方面

    ​ InnoDB写的效率高于MyISAM

    ​ MyISAM读的效率高于InnoDB

  • 从应用层面

    ​ InnoDB引擎主要面向在线事务处理方面的应用，MyISAM引擎主要面向在线分析方面的应用

  • 其他方面

    ​ 清空整张表时，InnoDB是一行一行的删除，而MyISAM是新建一张表

InnoDB引擎的优点体现在哪些方面，存储引擎如何科学合理地选择？

常见的Mysql存储引擎有InnoDB、MyISAM、MEMORY

​ InnoDB引擎是事务型数据库的首选引擎，支持支持事务，外键，行锁和MVCC(多版本并发控制)模式的读写，InnoDB是默认的MySQL引擎，InnoDB是为处理巨大数据量的最大性能设计，CPU效率极高

​ MyISAM引擎占用空间小，针对Insert，Select为主的应用访问速度块，支持全文索引，但不支持事务和行锁，并发性能较差，数据缓存靠操作系统的缓冲区实现，效率不佳

​ MEMORY的数据存储在内存而不是磁盘，表的生命周期比较短，通常用该引擎来建中间表

InnoDB 的四大特性是什么?

  • 插入缓冲(insert buffer)

    ​ 对于非聚集索引页的插入操作先进行缓存，以一定的频率，将多个插入操作合并为一个，提升性能

  • 二次写(double write)

    ​ 将数据的副页写入共享表，防止数据库宕机导致的数据丢失，提升数据库的健壮性

  • 自适应哈希索引(ahi)

    ​ 对于可以通过建立哈希索引而提升速度的查询，InnoDB引擎会自动根据访问的频率和模式来为热点页建立哈希索引

  • 预读(read ahead)

    ​ 预测并加载磁盘页数据到缓存池的异步io操作，采用线性预读（临近页预读）、随机预读（buffer预读）两种算法

InnoDB 为何推荐使用自增主键？

​ 自增 ID 可以保证每次插入时 B+ 树索引是从右边扩展的，因此相比自定义 ID （如 UUID）可以避免 B+ 树的频繁合并和分裂。如果使用字符串主键和随机主键，会使得数据随机插入，效率比较差。

事务的四大特性(ACID)是什么？底层是如何实现的？

  • 原子性：事务是最小的执行单位，不可以分割

  • 一致性：事务执行前后，数据保持一致，多个事务读取结果相同

  • 隔离性：并发访问数据库时，一个事务不会被另一个事务所干扰

  • 持久性：事务提交后产生的数据变更是持久的，即使数据库发生错误

  • 原子性通过undo log（回滚日志）实现：

    ​ 当对数据库进行一次修改或者插入操作时，生成undo log记录sql执行信息，如果事务执行失败或调用了rollback，导致事务需要回滚，便可以利用undo log中的信息将数据回滚到修改之前的样子

  • 持久性通过redo log（重做日志）实现：

    ​ redo log是一种预日志，当数据刷入buffer池中时，先在redo log中记录这一次记录，当事务提交时，会调用fsync接口对redo log进行刷盘。如果MySQL宕机，重启时可以读取redo log中的数据，对数据库进行恢复。

  • 隔离性通过锁机制和MVCC模式实现：

    ​ 写操作对写操作的干扰：锁机制（排他锁）保证隔离性

    ​ 写操作对读操作的干扰：MVCC保证隔离性，MVCC如何解决事务并发问题

  • 一致性通过保证其他三个特性来实现：

    ​ 与此同时，数据库本身也提供保障，例如不允许向整形列插入字符串值、字符串长度不能超过列的限制等

事务有哪些并发问题？对于哪些事务的隔离级别？

  • 脏读：一个事务读取到另一个事务尚未提交的数据

  • 不可重复读：一个事务两次读取数据的内容不一致

  • 幻读：一个事务两次读取数据的数据量不一致

事务的隔离级别：

Mysql默认的隔离级别为Repeatable read

锁的分类？

  • 按粒度分：

    ​ 页级锁、表级锁、行级锁

  • 按性质分：

    ​ 共享锁（s）、排他锁（x）、更新锁（u）

  • 按主观上分：

    ​ 乐观锁、悲观锁

隔离级别和锁的关系？

  • Read Uncommitted :

读取数据不需要加共享锁，这样就不会跟被修改的数据上的排他锁冲突

  • Read Committed :

    读操作需要加共享锁，但是在语句执行完以后释放共享锁

  • Repeatable Read :

    读操作需要加共享锁，等待事务执行完毕以后才释放共享锁；

  • SERIALIZABLE :

    限制性最强，因为该级别锁定整个范围的键，并一直持有锁，直到事务完成。

MySQL中有哪些常见的日志

  • redo log（重做日志）：保障事务持久性
  • undo log（回滚日志）：保证事务原子性
  • bin log（二进制日志）：用于主从同步和数据同步
  • error log（错误日志）：记录错误信息
  • general query log（普通查询日志）：记录每一条sql，开销较大，默认关闭
  • slow qurey log（慢查询日志）：记录执行时间过长和没有使用索引且执行成功的查询语句
  • relay log（中继日志）：在从节点中存储接收到的 binlog 日志内容，用于主从同步

什么是主从复制，有什么作用，有哪些架构形式？

概念：

​ 主从复制是指建立一个与主数据库完全一样的数据库环境，即从数据库

作用：

  • 做数据的热备:

    作为后备数据库，主数据库服务器故障后，可切换到从数据库继续工作，避免数据丢失

  • 架构的扩展:

    业务量越来越大，I/O访问频率过高，单机无法满足，此时做多库的存储，降低磁盘I/O访问的频率，提高单个机器的I/O性能。

  • 可以实现读写分离:

    使数据库能支撑更大的并发。在报表中尤其重要。由于部分报表sql语句非常的慢，导致锁表，影响前台服务。如果前台使用master，报表使用slave，那么报表sql将不会造成前台锁，保证了前台速度。

主从复制的实现原理是什么？

​ 数据库有个 binlog 二进制文件，记录了数据可执行的所有 SQL 语句。主从同步的目标就是把主数据库的 binlog 文件中的 SQL 语句复制到从数据库，让其在从数据库的 relaylog 文件中再执行一次这些 SQL 语句即可。
